# common-rules.md — 전 프로젝트 공통 작업 규칙

> 이 문서는 모든 프로젝트의 AGENTS.md에서 참조하는 공통 규칙이다.
> 프로젝트별 고유 규칙은 각 AGENTS.md에 있다.
> MUST/MUST NOT은 RFC 2119 의미로 사용한다.

---

## 1. 트랙 판단

### 트랙 A — 즉시 진행

조건 (모두 충족):
- 요청이 명확하다 (무엇을 어떻게 바꿀지 특정 가능)
- 해법이 하나뿐이다 (선택지/트레이드오프 없음)
- 영향 범위가 좁다 (1~2개 파일, 고위험 함수 미포함)

→ 영향 범위 분석 후 바로 작업지시서 출력.

### 트랙 B — 방향 확인 후 진행

조건 (하나라도 해당):
- 해법이 여러 개이고 선택이 필요하다
- 요청이 모호하거나 의도 확인이 필요하다
- 영향 범위가 넓다 (3개+ 파일, 고위험 함수 포함)
- 기존 동작이 바뀔 수 있다

→ 방향 확인서 출력 → 사용자 승인 → 작업지시서 출력. 승인 후 재확인하지 않는다.

### 판단 규칙
- 트랙 A면 `[트랙 A]`, 트랙 B면 `[트랙 B]` 표기
- 판단이 애매하면 트랙 B
- 사용자가 "바로 만들어" 등 즉시 진행을 명시하면 트랙 A로 전환

### 방향 확인서 형식
요청 이해 / 원인 분석(버그 시) / 해결 방향 / 영향 범위 / 관련 규칙 / 대안

## 2. 작업지시서 형식

⚠️ 모든 Step을 빠짐없이 순서대로 실행하세요. 특히 마지막 커밋 Step을 절대 생략하지 마세요.

### 필수 섹션
- 고정 문구: ⚡즉시 실행(질문 금지) / 프로젝트 경로(절대) / 기존 파일 전제(새 파일 시 제거)
- 작업 유형: 기능 추가 / 버그 수정 / 정리
- 영향 범위 분석: 수정 대상/금지, 전역 변수/함수, 고위험 함수, POLT 검증(① Dry Run ② 교훈 ③~⑤ 결함), 변경 후 기존 동작
- Step별: 파일(절대 경로) / 위치(함수명) / 작업 / 교체 코드(함수 전체) / 영향 함수 / 완료 확인
- Step N: 커밋 & 푸시
- ⛔종료(추가 작업·질문 금지)
- 최종 확인 (사용자 수동) — 핵심 1~2개 + 회귀 1개, 합계 3개 이하. 형식: `[ ] [동작] → [기대 결과] (핵심/회귀)`

### 작업지시서 규모 제한
- 커밋 Step 포함 10 Step 초과 시 작업지시서를 분할한다.
- 분할 시 각 작업지시서에 독립된 CHANGELOG Step과 커밋 Step을 포함한다. 이전 작업의 커밋을 확인한 뒤 다음 작업지시서를 전달한다.

## 3. 작업 유형별 규칙

**기능 추가**: 새 함수 전에 기존 함수 재사용 가능 여부 확인. 새 CSS 추가 시 같은 선택자 존재 여부 먼저 검색. !important 금지.
**버그 수정**: 수정 전에 원인을 먼저 설명. 새 규칙 덮어쓰기 금지. 잘못된 선언을 직접 수정.
**정리(리팩토링)**: 동작 변경 없음을 확인한다.

#### 데이터 스키마 변경
LocalStorage·GAS DB의 필드 추가/삭제/이름 변경 시:
- 신규 필드: 기존 데이터에 fallback 기본값 제공 (예: `field ?? defaultValue`)
- 삭제/이름 변경: 마이그레이션 함수 작성, 앱 시작 시 1회 실행, 멱등성 보장
- 변경 내역을 CHANGELOG에 기록

## 4. 갱신 규칙

### AGENTS.md
코드 구조를 기재하지 않으므로 코드 변경에 따른 갱신은 불필요.
갱신 필요: 새 파일 추가 시 파일 구조에 추가, 고위험/중위험 함수 목록 변경 시, 운영 규칙 자체 변경 시.

### CLAUDE.md
AGENTS.md 고유 주의사항이 변경되면 해당 프로젝트 CLAUDE.md도 갱신한다. 작업지시서에 AGENTS.md 갱신 Step이 포함될 때 CLAUDE.md 갱신 필요 여부를 확인하고, 필요하면 Step을 추가한다.

### 운영 문서 갱신
모든 작업지시서의 마지막 커밋 Step 직전에 CHANGELOG.md 갱신 Step을 포함한다. 절대 생략하지 않는다.
- CHANGELOG는 모든 작업지시서에서 필수 갱신
- opus.md 파일: C:\dev\opus\opus.md
- opus.md 크롤링: https://raw.githubusercontent.com/leftjap/opus/main/opus.md
- opus.md 갱신 불필요: 사소한 수정(CSS만 변경, 오타 수정 등). CHANGELOG는 이 경우에도 기록한다.

### CHANGELOG 갱신

모든 작업지시서의 마지막 커밋 Step 직전에 해당 프로젝트의 CHANGELOG.md를 갱신한다.

- 형식: [Keep a Changelog](https://keepachangelog.com/en/1.1.0/) — Added / Changed / Fixed / Removed
- 날짜 헤더: `## YYYY-MM-DD`. 같은 날짜 헤더가 이미 있으면 해당 섹션에 추가. 없으면 파일 최상단(기존 날짜 위)에 새 헤더 생성
- 항목 형식: `- 한 줄 요약 — 부연 설명. (변경 파일)`
- 큰 결정(아키텍처·스키마·규칙 체계 변경)은 Changed 항목에 이유 1문장을 덧붙인다
- Fixed 항목에 원인 태그를 `[영역.세부원인]` 형식으로 표기한다. 동일 원인의 수정이면 같은 태그를 사용한다.
  - 영역 (고정 목록, 필요시 추가): `로직` · `상태` · `경로` · `UI` · `환경` · `연동` · `설정`
  - 세부원인: 2~4글자 한글 키워드 (예: 파싱순서, 초기화누락, 캐시, 세이프영역, 빌드산출물)
  - 예시: `- 메타 섹션 파싱 오염 수정 [로직.파싱순서] (parser.js, renderer.js)`
- CHANGELOG.md가 크롤링 한계(10,000 토큰)에 근접하면 현재 연도분만 유지하고 이전 연도를 `CHANGELOG-{연도}.md`로 분리한다

### GAS 배포
Code.js 수정 시 반드시 clasp push Step을 포함한다.
- MUST: 기존 배포를 업데이트한다 (@HEAD 또는 기존 배포 ID에 새 버전 지정). URL이 유지된다.
- MUST NOT: 새 배포(Deploy → New deployment)를 생성한다 — URL이 변경되어 모든 클라이언트의 GAS_URL 수정이 필요해진다.
- MUST: GAS 배포 성공 후 Git 커밋. GAS 실패 시 클라이언트 코드도 푸시하지 않는다.
- MUST NOT: 작업지시서에 "Google Apps Script 에디터에서 수동 재배포" 안내를 포함한다. GitHub Actions auto-deploy(B-63)가 처리한다.

#### 배포 후 검증
clasp push 완료 후, 해당 프로젝트의 기존 GET 액션(예: list_books, load_db)을 1회 호출하여 HTTP 200 응답을 확인한다. 실패 시 커밋하지 않고 사용자에게 보고한다.

#### 롤백 절차
1. `clasp versions` 로 이전 버전 번호 확인
2. `clasp deploy -i {deploymentId} -V {이전버전번호} -d "rollback"`
3. 관련 git 커밋을 `git revert` 로 되돌림

※ 현재 프로젝트는 clasp v2.x 기준이다. v3.x 전환 시 `clasp list-versions` → `clasp create-deployment --deploymentId {id} -V {ver}` 로 변경한다.

## 5-1. 코드 품질 사전 검증 — POLT (Pre-Output Logic Trace)

> 작업지시서에 코드 블록을 포함하기 전에 수행하는 검증. 결과를 영향 범위 분석에 필수 출력.

### 적용 조건
작업지시서 Step이 코드를 생성·교체·수정할 때.

코드 변경 규모와 무관하게 수행한다. 작업지시서마다 독립 수행한다.

### 절차

| # | 단계 | 핵심 행동 |
|---|---|---|
| ⓪ | Scope Lock | 변경 파일·함수·변수·CSS 나열. 호출자 역추적(시그니처·반환값·공유 상태 변경 시). 변경 금지 영역 나열(AGENTS.md ⛔ Freeze · 🛡️ Guard 보호 체크리스트 자동 포함). 이후 ①~⑤는 변경 대상만을 범위 |
| ① | Dry Run | 대표 입력 1~2개, 라인별 변수 추적, 모든 분기 1회+ 통과, 루프 내 I/O 성능 표시. 새 테스트 생성 시, Step 0으로 대상 함수에 SAMPLE을 실행해 actual을 콘솔 출력한 뒤 그 값을 expected로 사용한다 |
| ② | Self-Critique | 교훈 L-xx 전수 대조(의무) → AGENTS.md 체크리스트 대조 → 엣지케이스(null, undefined, 빈 배열, 빈 문자열, 0, 음수, 경계값) → 환각 객체 → 보안 → 중복 → 과잉 방어 → 유지보수성 |
| ③ | Wiring Check | 새 요소의 호출·참조 경로 확인. 기존 caller 동작 보존 확인. 공유 상태(전역, DOM, CSS, LS, GAS) 영향 확인. 크롤링 제외 caller는 ⚠️ 표기 |
| ④ | Preservation | 변경 금지 영역(⓪)과 실제 변경 교차 검증. 기존 동작이 달라지는 경로 추적 |
| ⑤ | Faithfulness | ①~④ 결함이 최종 코드에 반영되었는지 확인 |
| ⑥ | Step Sequence | 전체 Step 역순 정합성. 새 요소 연결·해제 확인. 최종 1회만 |

### 강화 조건
교훈 해당·3개+ 파일·고위험 함수·스키마 변경·같은 문제 2회+ 시, 교차 영향 시뮬레이션과 변경 전후 비교를 추가 수행한다. 사용자가 "꼼꼼하게"·"시뮬레이션" 지시 시 롤백 시나리오도 포함한다. 강화 조건 해당 시, 최종 확인 항목에 구체적 입력값과 기대 결과를 명시한다. 항목 수 상한(3개)은 유지하되, POLT에서 가장 높은 리스크로 판단한 경로를 우선 배치한다(confident-but-wrong 방지).

### 출력 규칙
영향 범위 분석에 필수 포함:
- ① Dry Run: [입력] → [추적 요약] → [결과]
- ② 교훈 대조: [해당 L-xx → 반영] 또는 [대조 범위: L-xx~L-xx, 해당 없음]. 체크리스트: [해당 항목 → 반영] 또는 [해당 없음]
- ③~⑥: 각 단계별 [점검 대상 1개+] → 결함 없음 / [결함 내용]. 점검 대상을 생략하지 않는다.

### 반복 규칙
결함 발견 시 코드 수정 후 ①~⑤ 재수행. ⑥은 최종 1회. **최대 2회 반복.** 3회차에도 잔존 시 `⚠️ POLT 미해결 — [내용]` 표기.

## 6. 소스 참조 공통 규칙

### 참조 우선순위
1순위 — 사용자 업로드 파일 (push 안 한 변경분 우선)
2순위 — GitHub raw URL 크롤링
3순위 — 사용자에게 업로드 요청 (1, 2 불가 시에만)

### 크롤링 규칙
- 소스 크롤링 전에 AGENTS.md의 "크롤링 제외" 목록을 확인한다. 제외 파일은 크롤링하지 않고 업로드를 요청한다.
- 크롤링 실패(잘림/빈 응답) 시: 사용자에게 업로드를 요청한다. 잘린 파일 기반으로 작업지시서를 작성하지 않는다.
- 크롤링 제외 목록의 관리는 수동 일괄 점검으로 수행한다. 크롤링 실패 시 자동 등록하지 않는다.
- 한 세션 최대 4파일

### 혼합 참조·컨텍스트
직전 수정 파일·잘린 파일은 크롤링 금지, 업로드 요청. 컨텍스트 압축 시 사용자 알림 + GitHub raw 재확인(기억 의존 금지).

## 7. 문서 관리 규칙 (opus.md에서 이관)

### 작업환경 함의

opus.md 섹션 0-1에 요약되어 있다. 핵심: 작업지시서는 Haiku가 읽으므로 Opus 수준 추론을 가정하지 말 것.

### Progressive Disclosure 상세

**대전제:** 사용자는 세션 시작 시 opus.md만 업로드한다. 나머지 모든 운영 문서는 GitHub raw URL로 크롤링 가능해야 한다.

| 규칙 | 기준 |
|---|---|
| 신규 문서 | GitHub 레포에 배치. 상위 문서(AGENTS.md 또는 opus.md)에 참조 URL 기재 |
| AGENTS.md | 200줄 이하. 초과 시 보조 문서로 분리하고 URL만 남김 |
| opus.md | 200줄 이하. 교훈 20건 초과 시 아카이브, 완료 백로그는 backlog-archive.md로 이관 |
| 크롤링 한계 초과 (10,000 토큰) | AGENTS.md 소스 참조에 "크롤링 제외 — 업로드 필요" 명시 |
| 문서 역할 혼재 금지 | Opus용 문서(AGENTS.md)에 Haiku 전용 내용을 넣지 않음. 역도 동일 |
| 크롤링 제외 파일 | 해당 AGENTS.md 소스 참조에 기록. 수동 일괄 점검으로 관리 |

## 8. 로컬 백업

로컬 백업: `C:\dev\backups\{프로젝트명}\` (Git 외부).

## 9. 프로세스 안전 규칙 (Process Safety)

1. **프로세스 종료 명령 사용 금지**: `Stop-Process`, `taskkill`, `kill`, `pkill` 등 프로세스 종료 명령을 작업지시서 및 실행 스크립트에 포함하지 않는다.
2. **GUI 앱 실행/종료는 사용자 수동**: GUI 실행파일의 시작과 종료는 사용자가 직접 수행한다. 작업지시서에는 `사용자: [앱 이름]을 종료해주세요` 형태의 안내문만 포함한다.
3. **시스템 프로세스 이름 충돌 주의**: 빌드 출력물 이름이 시스템 프로세스명(explorer, svchost, csrss, dwm, lsass 등)과 동일한 경우, 프로세스 관련 명령을 절대 사용하지 않는다.
4. **빌드 전 앱 종료 필요 시**: `사용자: dist 폴더의 앱이 실행 중이면 종료해주세요`라고 안내하고, 사용자 확인 후 빌드를 진행한다.
