# 설계 편향 방지 규칙(D-xx) 도입

- date: 2026-03-28
- tags: playbook, opus, bias, design-rules

## 맥락

playbook.md에 Haiku 실행 실패 방지 교훈(L-xx)은 정비되어 있었으나, Opus 설계 단계의 사고 편향을 견제하는 규칙이 없었다. 사용자가 이 부재를 지적.

## 조사 근거

- Knipper et al. (2025): 모든 LLM이 17.8%~57.3%의 인지 편향 취약률. 아첨(sycophancy)은 RLHF 구조적 산물.
- ceaksan.com (60편+ 논문 종합): LLM 행동 실패 모드 분류. 아첨, 앵커링, 확증 편향, 프레이밍이 설계 AI 역할에서 고위험.
- Claude Code 시스템 프롬프트 역설계 (Feng Liu, 2026): "정확성 > 사용자 동의" 우선순위 규칙이 톤 규칙("아첨하지 마")보다 효과적.
- USC 연구가이드: 프레이밍 방지는 체크리스트가 아니라 워크플로우 첫 단계로 넣어야 instruction attenuation에 견딤.

## 시뮬레이션 결과

백로그 4건(B-02 판정UI, B-15 댓글보강, B-38 persist, I-05 React)으로 D-xx 적용 전/후를 비교. D-04(양비론 회피)와 D-05(프레이밍 검증)가 1인 운영 환경에서 가장 임팩트 큼.

## 시뮬레이션 후 수정

- D-02: "업계 표준이나 기존 합의가 아닌 한" 예외 추가. 업계 컨벤션 수치까지 매번 확인하면 불필요한 마찰.
- D-04: "순수 취향은 예외" 추가. 미적/취향 선택에서 억지 추천 방지.
