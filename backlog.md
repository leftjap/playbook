# 백로그

> opus.md에서 분리된 백로그. 등록·읽기·관리 규칙은 opus.md 섹션 2 참조.

---

## 등록 규칙

### 등록일

- 모든 백로그 항목은 등록 시점의 날짜·시간을 `MM/DD HH:MM` (KST) 형식으로 기록한다.
- 테이블의 `등록일` 컬럼과 상세 블록의 `- **등록일:**` 필드에 동일한 값을 기입한다.
- 등록 주체(AI)가 등록 시점에 자동으로 기입한다. 사용자가 직접 기입하지 않는다.
- 병합된 항목은 가장 오래된 원본의 등록일을 사용한다.
- 등록일은 1회 기입 후 변경하지 않는다.

### 빠른 등록

사용자가 "백로그에 저장/등록/추가"를 지시하면:
1. 해결·분석을 시도하지 않는다. 기록만 한다.
2. 기록 항목: ①프로젝트명 ②한 줄 요약(사용자 표현을 풀어쓴 1~2문장) ③대화 맥락(어떤 작업 중에, 무엇을 하다가 나온 건지)
3. 상태: ⚪ 메모. 착수 시점에 아래 등록 규칙(완료 조건, 관련 코드, 버그 시 재현·원인 추정)을 보완하여 🟡 이상으로 전환한다.

### 착수 가능 항목 등록 규칙

1. 대화에서 백로그 항목이 **언급·추가·변경**될 때마다 즉시 상세 블록을 기록한다. "나중에 채운다"는 금지. 단, 증상이나 요구사항이 불명확하여 한 줄 요약조차 작성할 수 없는 경우, 사용자에게 확인한 뒤 등록한다.
2. 모든 항목은 상태(🔴/🟡/⚪)와 무관하게 **상세 블록을 처음부터** 갖는다.
3. **한 줄 요약**은 사용자 언어로 쓴다. 함수명·변수명 등 기술 용어를 넣지 않는다.
4. **완료 조건**은 사용자 행동 기반으로 쓴다. "사용자가 ~하면 ~된다" 또는 "~화면에서 ~가 보인다" 형식.
5. 완료된 항목은 `backlog-archive.md`로 이동한다.
5-1. 사용자가 명시적으로 완료·취소·보류를 선언한 항목도 이관한다. 사유를 "현재" 필드에 기록한다.
6. **모든 항목에 "관련 코드" 필드를 채운다.** 등록 시점에 소스를 보고 있으므로 관련 파일·함수·흐름을 기록한다. 다음 세션의 Opus가 이 필드만 보고 어떤 파일을 크롤링해야 하는지 판단할 수 있어야 한다.
6-1. 관련 코드가 "미확인"인 항목은 해당 프로젝트의 AGENTS.md 크롤링을 완료한 뒤 관련 코드를 갱신해야 작업에 착수할 수 있다.
7. **버그 항목은 "재현" + "원인 추정" 필드를 필수로 채운다.** 재현의 "관찰"은 스크린샷을 대체하는 텍스트 묘사다 — 화면에 보이는 것을 그대로 쓴다. 원인 추정은 ①의심 코드 경로 ②확인한 것과 배제한 것 ③다음 확인 단계를 기록한다.
7-1. 버그 등록 시 재현 경로·환경·빈도를 확인할 수 없으면, 아는 필드만 채우고 모르는 필드에 "미확인 — 사용자 확인 필요"라고 기재한다. 이 상태의 버그는 작업 착수 금지이며, 사용자 확인 후 갱신한 뒤에만 착수한다.
8. **원인 추정이 "미분석"인 버그는 등록하지 않는다.** 등록 시점에 최소 1단계 분석을 수행하여 ①②③을 채운다. 다음 세션의 Opus가 ③부터 이어갈 수 있어야 한다. **예외: ⚪ 메모(빠른 등록)는 원인 추정 없이 등록 가능하며, 🟡 이상 전환 시 보강한다.**
