# common-rules.md — 전 프로젝트 공통 작업 규칙

> 이 문서는 모든 프로젝트의 AGENTS.md에서 참조하는 공통 규칙이다.
> 프로젝트별 고유 규칙은 각 AGENTS.md에 있다.
> MUST/MUST NOT은 RFC 2119 의미로 사용한다.

---

## 1. 트랙 판단

### 트랙 A — 즉시 진행

조건 (모두 충족):
- 요청이 명확하다 (무엇을 어떻게 바꿀지 특정 가능)
- 해법이 하나뿐이다 (선택지/트레이드오프 없음)
- 영향 범위가 좁다 (1~2개 파일, 고위험 함수 미포함)

→ 영향 범위 분석 후 바로 작업지시서 출력.

### 트랙 B — 방향 확인 후 진행

조건 (하나라도 해당):
- 해법이 여러 개이고 선택이 필요하다
- 요청이 모호하거나 의도 확인이 필요하다
- 영향 범위가 넓다 (3개+ 파일, 고위험 함수 포함)
- 기존 동작이 바뀔 수 있다

→ 방향 확인서 출력 → 사용자 승인 → 작업지시서 출력. 승인 후 재확인하지 않는다.

### 판단 규칙
- 트랙 A면 `[트랙 A]`, 트랙 B면 `[트랙 B]` 표기
- 판단이 애매하면 트랙 B
- 사용자가 "바로 만들어" 등 즉시 진행을 명시하면 트랙 A로 전환

### 방향 확인서 형식

## 방향 확인: [요청 요약]
### 요청 이해 — [1~3문장]
### 원인 분석 (버그 수정 시) — [어디서 어떤 값이 적용되어 이런 결과가 나오는지]
### 해결 방향 — [어떤 파일의 어떤 함수를 어떻게 바꿀 것인지]
### 영향 범위 — [영향 받는 함수/변수/플랫폼별 차이]
### 관련 기존 규칙 — [참조할 규칙 번호. 없으면 "없음"]
### 대안 (있을 경우) — [장단점과 함께]

## 2. 작업지시서 형식

⚠️ 모든 Step을 빠짐없이 순서대로 실행하세요. 특히 마지막 커밋 Step을 절대 생략하지 마세요.

### 필수 섹션
- 프로젝트 경로 (절대 경로)
- "모든 파일은 이미 존재합니다. 새로 만들지 마세요." (새 파일 시 제거)
- 작업 유형: 기능 추가 / 버그 수정 / 정리
- 영향 범위 분석: 수정 대상 파일, 수정 금지, 영향 받는 전역 변수/함수, 고위험 함수 수정 여부, 변경 후 확인할 기존 동작
- Step별: 파일(절대 경로), 위치(함수명), 작업, 교체 코드(함수 전체), 영향 받는 함수, 완료 확인
- Step N-1: playbook.md 갱신 (해당 시)
- Step N: 커밋 & 푸시
- ⛔ 여기서 작업을 종료하세요.
- 최종 확인 (사용자 수동)

## 3. 작업 유형별 규칙

**기능 추가**: 새 함수 전에 기존 함수 재사용 가능 여부 확인. 새 CSS 추가 시 같은 선택자 존재 여부 먼저 검색. !important 금지.
**버그 수정**: 수정 전에 원인을 먼저 설명. 새 규칙 덮어쓰기 금지. 잘못된 선언을 직접 수정.
**정리(리팩토링)**: 전후 동작 동일. 파일 하나씩 진행.

## 4. Claude Code 실행 규칙

- MUST: 한 Step에 한 파일, 한 가지 변경
- MUST: 함수 수정 시 함수 전체를 교체 코드로 제공. 부분 스니펫 비교 금지
- MUST: CSS 수정 시 선택자 블록 전체 제공
- MUST: 모든 파일 경로는 절대 경로
- MUST: 기존 함수 수정 시 함수명과 현재 동작을 Step에 명시
- MUST: 각 Step에 완료 확인 조건 포함
- MUST: 마지막 Step에 커밋 & 푸시
- MUST: 커밋 메시지 형식 `[타입]: [요약]` (feat/fix/chore/refactor)
- MUST: 영향 범위 분석에 "수정 대상 파일"과 "수정 금지" 명시
- MUST: "변경 후 확인할 기존 동작" 체크리스트 포함
- MUST NOT: "적절히", "비슷하게" 등 모호한 표현
- MUST NOT: sed, tr 등 Unix 텍스트 처리 도구 (Windows 환경)
- MUST NOT: 부분 스니펫 비교 방식

### 반복 실패 방지
- 동일 증상 fix/style 커밋 2회 연속 실패 시 3회차 전 사용자에게 경고: "⚠️ 컨텍스트 오염 확률이 높습니다. 다른 세션에서 시도해 보세요."
- 3회 연속 실패 시 즉시 중단, 시도 내역·실패 원인·다음 방향을 정리하여 보고

## 5. 갱신 규칙

### AGENTS.md
코드 구조를 기재하지 않으므로 코드 변경에 따른 갱신은 불필요.
갱신 필요: 새 파일 추가 시 파일 구조에 추가, 고위험/중위험 함수 목록 변경 시, 운영 규칙 자체 변경 시.

### playbook.md
모든 작업지시서의 커밋 & 푸시 Step 직전에 playbook.md 갱신 Step을 포함한다. 절대 생략하지 않는다.
- 파일: C:\dev\playbook\playbook.md
- 크롤링: https://raw.githubusercontent.com/leftjap/playbook/main/playbook.md
- 갱신 불필요: CSS만 변경, 오타 수정 등 백로그 외 사소한 수정

### GAS 배포
Code.js 수정 시 반드시 clasp push Step + 웹앱 재배포 안내 포함.
GAS 배포 성공 후 Git 커밋. GAS 실패 시 클라이언트 코드도 푸시하지 않는다.

## 6. AI 응답 규칙

- 작업 규모를 부풀리지 않는다. 한 번에 할 수 있으면 묻지 않고 한 번에 한다.
- 선택지를 나열하는 것으로 끝내지 않는다. 추천 + 근거를 붙인다.
- 확신 수준: "확실합니다" / "높은 확률이지만 검증 필요" / "추측입니다".
- 검증 강도 "강화" 이상에서는 코드에 문제가 있다고 가정하고 찾는다.
- 사용자에게 콘솔 명령 실행을 요청하기 전에 코드 정적 분석으로 원인을 먼저 추정한다.
- 에이전트 전달용 버그 브리핑에서 해법을 단정하지 않는다. 증상 + 추정 원인 + 확인 포인트까지 제공.

## 7. 코드 비대화 방지

- 렌더 함수 80줄 초과 시 하위 함수 분리
- 동일 로직 모바일/PC 함수는 platform 매개변수로 통합
- 새 기능 전에 유사 기존 함수 확인
- 새 CSS 전에 기존 클래스 재사용 가능 여부 확인

## 8. 파일 업로드 검증

사용자가 파일을 업로드하면 첫 줄의 `PROJECT:` 주석으로 프로젝트를 확인한다.
주석이 없으면 파일 내 핵심 함수/변수로 판별한다.
현재 작업 프로젝트와 불일치하면 즉시 중단하고 사용자에게 확인한다.

## 9. 디버깅 프로토콜

### 1단계 — AI 자체 해결 (사용자 개입 0)
파일 구조에서 관련 파일 특정 → GitHub raw URL로 소스 직접 읽기 → 가설 수립 → 수정 코드 특정 → 작업지시서 출력.

### 2단계 — 1회 요청 (사용자 개입 1회)
조건: 런타임 상태를 알 수 없을 때. 콘솔 명령어를 한 번에 전부 제시. 예상 결과 포함.

### 3단계 — 브라우저 조작 (최후 수단)

### 요청 규칙
- 소스코드로 먼저 추론. 추론으로 특정되면 스크린샷/콘솔 없이 작업지시서 출력.
- 콘솔 요청 금지: 함수 존재 여부, CSS 선택자, HTML 구조, 전역 변수 선언, 코드 로직.
- 콘솔 필요: LocalStorage 실제 데이터, 런타임 변수 값, computed style, JS 에러, 네트워크 결과.

## 10. 소스 참조 공통 규칙

### 참조 우선순위
1순위 — 사용자 업로드 파일 (push 안 한 변경분 우선)
2순위 — GitHub raw URL 크롤링
3순위 — 사용자에게 업로드 요청 (1, 2 불가 시에만)

### 크롤링 규칙
- 4개 이하: 사용자에게 묻지 않고 크롤링
- 5개 이상: 안내 + 동시에 크롤링 시작
- 한 세션 최대 4파일

### 혼합 참조 주의
- 직전 작업지시서가 수정한 파일은 크롤링 대신 업로드 요청
- 파일이 잘렸으면 업로드 요청

### 컨텍스트 압축 발생 시
- 사용자에게 알림 + 재업로드 요청
- 기억 의존 금지, GitHub raw로 재확인

## 11. 정보 참조 우선순위

AI가 작업에 필요한 정보를 모를 때 사용자에게 묻기 전에 아래 순서를 따른다.

1. playbook.md / 인수인계서
2. 해당 프로젝트 AGENTS.md
3. 직접 확인 — PowerShell, git, 크롤링 등으로 현재 상태를 직접 조회
4. 사용자에게 요청 — 위 3가지로 불가능할 때만
