# common-rules.md — 전 프로젝트 공통 작업 규칙

> 이 문서는 모든 프로젝트의 AGENTS.md에서 참조하는 공통 규칙이다.
> 프로젝트별 고유 규칙은 각 AGENTS.md에 있다.
> MUST/MUST NOT은 RFC 2119 의미로 사용한다.

---

## 1. 트랙 판단

### 트랙 A — 즉시 진행

조건 (모두 충족):
- 요청이 명확하다 (무엇을 어떻게 바꿀지 특정 가능)
- 해법이 하나뿐이다 (선택지/트레이드오프 없음)
- 영향 범위가 좁다 (1~2개 파일, 고위험 함수 미포함)

→ 영향 범위 분석 후 바로 작업지시서 출력.

### 트랙 B — 방향 확인 후 진행

조건 (하나라도 해당):
- 해법이 여러 개이고 선택이 필요하다
- 요청이 모호하거나 의도 확인이 필요하다
- 영향 범위가 넓다 (3개+ 파일, 고위험 함수 포함)
- 기존 동작이 바뀔 수 있다

→ 방향 확인서 출력 → 사용자 승인 → 작업지시서 출력. 승인 후 재확인하지 않는다.

### 판단 규칙
- 트랙 A면 `[트랙 A]`, 트랙 B면 `[트랙 B]` 표기
- 판단이 애매하면 트랙 B
- 사용자가 "바로 만들어" 등 즉시 진행을 명시하면 트랙 A로 전환

### 방향 확인서 형식

## 방향 확인: [요청 요약]
### 요청 이해 — [1~3문장]
### 원인 분석 (버그 수정 시) — [어디서 어떤 값이 적용되어 이런 결과가 나오는지]
### 해결 방향 — [어떤 파일의 어떤 함수를 어떻게 바꿀 것인지]
### 영향 범위 — [영향 받는 함수/변수/플랫폼별 차이]
### 관련 기존 규칙 — [참조할 규칙 번호. 없으면 "없음"]
### 대안 (있을 경우) — [장단점과 함께]

## 2. 작업지시서 형식

⚠️ 모든 Step을 빠짐없이 순서대로 실행하세요. 특히 마지막 커밋 Step을 절대 생략하지 마세요.

### 필수 섹션
- ⚡ 이 작업지시서를 받으면 즉시 순서대로 실행하세요. 실행 여부를 사용자에게 묻지 마세요.
- 프로젝트 경로 (절대 경로)
- "모든 파일은 이미 존재합니다. 새로 만들지 마세요." (새 파일 시 제거)
- 작업 유형: 기능 추가 / 버그 수정 / 정리
- 백로그 컨텍스트 (백로그 연결 작업일 때 필수):
  - 대상: B-xx (프로젝트 · 작업명)
  - 이 작업지시서가 완료하는 항목: (backlog.md 완료 조건 중 어떤 체크박스를 충족하는지)
  - 완료 후 backlog.md 갱신: (해당 체크박스 [x] 처리 + 현재 상태 문구 업데이트)
- 영향 범위 분석: 수정 대상 파일, 수정 금지, 영향 받는 전역 변수/함수, 고위험 함수 수정 여부, POLT 검증 (① Dry Run, ② 교훈 대조, ③~⑤ 결함), 변경 후 확인할 기존 동작
- Step별: 파일(절대 경로), 위치(함수명), 작업, 교체 코드(함수 전체), 영향 받는 함수, 완료 확인
- Step N-1: opus.md 갱신 (해당 시) — 파일: `C:\dev\opus\opus.md`. 다른 레포에서 실행 중일 때 절대 경로 필수.
- Step N: 커밋 & 푸시
- ⛔ 여기서 작업을 종료하세요. 추가 작업을 수행하거나 사용자에게 다음 행동을 묻지 마세요.
- 최종 확인 (사용자 수동) — Opus가 아래 형식으로 채운다. 항목 수: 핵심 1~2개 + 회귀 1개. 비대화 방지를 위해 합계 3개 이하.
  - [ ] [동작] → [기대 결과]  (핵심)
  - [ ] [동작] → [기대 결과]  (회귀 — 영향 범위 분석의 "변경 후 확인할 기존 동작"에서 선정)

## 3. 작업 유형별 규칙

**기능 추가**: 새 함수 전에 기존 함수 재사용 가능 여부 확인. 새 CSS 추가 시 같은 선택자 존재 여부 먼저 검색. !important 금지.
**버그 수정**: 수정 전에 원인을 먼저 설명. 새 규칙 덮어쓰기 금지. 잘못된 선언을 직접 수정.
**정리(리팩토링)**: 동작 변경 없음을 확인한다.

## 4. Claude Code 실행 규칙

- MUST: 한 Step에 한 파일, 한 가지 변경
- MUST: 함수 수정 시 함수 전체를 교체 코드로 제공. 부분 스니펫 비교 금지
- MUST: CSS 수정 시 선택자 블록 전체 제공
- MUST: 모든 파일 경로는 절대 경로
- MUST: 기존 함수 수정 시 함수명과 현재 동작을 Step에 명시
- MUST: 각 Step에 완료 확인 조건 포함
- MUST: 마지막 Step에 커밋 & 푸시
- MUST: 커밋 메시지 형식 — 백로그 연결 작업: `B-xx: [타입]: [요약]` / 백로그 무관 작업: `[타입]: [요약]` (feat/fix/chore/refactor)
- MUST: 영향 범위 분석에 "수정 대상 파일"과 "수정 금지" 명시
- MUST: "변경 후 확인할 기존 동작" 체크리스트 포함
- MUST: 백로그 연결 작업의 마지막 커밋 전에 backlog.md의 해당 체크박스를 [x]로 갱신하고, "현재" 상태 문구를 업데이트한다
- MUST: 백로그 항목의 모든 완료 조건이 충족되면, 해당 항목을 backlog-archive.md로 이동한다
- MUST: "최종 확인 (사용자 수동)" 섹션이 비어 있으면 안 된다. Opus가 작업지시서 생성 시 채운다
- MUST NOT: "적절히", "비슷하게" 등 모호한 표현
- MUST NOT: sed, tr 등 Unix 텍스트 처리 도구 (Windows 환경)

### 반복 실패 방지
- 동일 증상 fix/style 커밋 2회 연속 실패 시 3회차 전 사용자에게 경고: "⚠️ 컨텍스트 오염 확률이 높습니다. 다른 세션에서 시도해 보세요."
- 3회 연속 실패 시 즉시 중단, 시도 내역·실패 원인·다음 방향을 정리하여 보고

## 5. 갱신 규칙

### AGENTS.md
코드 구조를 기재하지 않으므로 코드 변경에 따른 갱신은 불필요.
갱신 필요: 새 파일 추가 시 파일 구조에 추가, 고위험/중위험 함수 목록 변경 시, 운영 규칙 자체 변경 시.

### CLAUDE.md
AGENTS.md 고유 주의사항이 변경되면 해당 프로젝트 CLAUDE.md도 갱신한다. 작업지시서에 AGENTS.md 갱신 Step이 포함될 때 CLAUDE.md 갱신 필요 여부를 확인하고, 필요하면 Step을 추가한다.

### opus.md
모든 작업지시서의 커밋 & 푸시 Step 직전에 opus.md 갱신 Step을 포함한다. 절대 생략하지 않는다.
- 파일: C:\dev\opus\opus.md
- 크롤링: https://raw.githubusercontent.com/leftjap/opus/main/opus.md
- 갱신 불필요: CSS만 변경, 오타 수정 등 백로그 외 사소한 수정

### CHANGELOG 갱신

모든 작업지시서의 opus.md 갱신 Step에서 프로젝트별 CHANGELOG.md도 갱신한다.

- 형식: [Keep a Changelog](https://keepachangelog.com/en/1.1.0/) — Added / Changed / Fixed / Removed
- 날짜 헤더: `## YYYY-MM-DD`. 같은 날짜 헤더가 이미 있으면 해당 섹션에 추가. 없으면 파일 최상단(기존 날짜 위)에 새 헤더 생성
- 항목 형식: `- 한 줄 요약 — 부연 설명. (변경 파일)` 또는 `- 한 줄 요약 (B-xx)`
- 큰 결정(아키텍처·스키마·규칙 체계 변경)은 Changed 항목에 이유 1문장을 덧붙인다
- 제외: opus.md/backlog.md 자체의 교훈·상태 갱신은 CHANGELOG에 기록하지 않는다
- CHANGELOG.md가 크롤링 한계(10,000 토큰)에 근접하면 현재 연도분만 유지하고 이전 연도를 `CHANGELOG-{연도}.md`로 분리한다

### GAS 배포
Code.js 수정 시 반드시 clasp push Step + 웹앱 재배포 안내 포함.
GAS 배포 성공 후 Git 커밋. GAS 실패 시 클라이언트 코드도 푸시하지 않는다.

## 6-1. 코드 품질 사전 검증 — POLT (Pre-Output Logic Trace)

> Opus가 작업지시서에 코드 블록을 포함하기 전에 내부적으로 수행하는 검증 절차.
> 검증 결과를 작업지시서의 영향 범위 분석 섹션에 필수 출력한다. 항목이 없는 작업지시서는 POLT 미수행으로 간주한다.
> 이 절차는 Extended Thinking / Think Tool의 구조화된 활용이다.

### 적용 조건
작업지시서의 Step이 코드를 생성·교체·수정할 때마다 해당 Step의 코드에 대해 수행한다.

### ⓪ Scope Lock — 변경 범위 확정
- 이번 Step에서 변경하는 파일·함수·변수·CSS 선택자를 나열한다.
- **호출자 역추적**: 시그니처·반환값·공유 상태를 변경하는 경우, AGENTS.md 파일 목록에서 해당 함수의 호출자(caller)를 역추적한다. 호출자가 작업지시서 Step에 포함되어 있지 않으면 Step을 추가하거나, 영향이 없음을 확인하여 기록한다.
- **변경 금지 영역**을 나열한다: 이번 Step에서 건드리지 않아야 할 파일·함수·전역 변수.
- AGENTS.md에 수정 금지(⛔)로 표시된 파일·함수는 자동으로 변경 금지 영역에 포함한다.
- 이후 ①~⑤는 변경 대상만을 범위로 한다. Scope 외 수정은 금지.

### ① Dry Run — 구체적 값 추적
- 대표 입력 1~2개를 선정한다.
- 생성한 코드를 라인별로 실행하듯 추적하며 변수 상태를 구체적 값으로 기록한다.
- if/else 분기, 루프는 각 경로를 최소 1회 통과시킨다.
- 루프 내부에 API 호출·디스크 I/O가 있으면 성능 위험으로 표시한다.

### ② Self-Critique — 체크리스트 검증
아래 질문에 순서대로 답한다.
1. **교훈 목록 (L-01 ~ L-xx)**: opus.md의 교훈 항목을 하나씩 대조하여 위반 여부를 확인한다. (의무)
2. **엣지케이스**: null, undefined, 빈 배열, 빈 문자열, 0, 음수, 경계값에서의 동작은?
3. **환각 객체**: 호출하는 함수·변수·API·CSS 클래스가 실제로 존재하는가?
4. **보안**: 하드코딩된 자격증명, 민감 데이터 노출, XSS 가능성이 있는가?
5. **중복**: 이미 동일 기능을 수행하는 기존 함수·CSS가 있는가?
6. **과잉 방어**: 추가한 방어 코드(null check, try-catch)가 근본 원인을 가리는가?
7. **유지보수성**: 3개월 뒤 이 코드를 수정해야 할 때 걸림돌이 되는 부분은?

### ③ Wiring Check — 의존성·통합 확인
- 새 함수·변수·CSS 클래스가 실제로 호출·참조되는 곳을 특정한다.
- 호출 경로가 없으면 (created-but-unconnected) 작업지시서에 연결 Step을 추가한다.
- 변경한 함수를 호출하는 **기존 caller 목록**을 나열하고, 각 caller의 동작이 보존되는지 확인한다. 크롤링 제외 파일이 caller에 포함되면 작업지시서에 ⚠️를 표기하고, 해당 파일의 업로드를 요청하거나 호출 부분의 교체 코드를 포함한다.
- 공유 상태 (전역 변수, DOM, CSS, LocalStorage, GAS PropertiesService)에 대한 영향을 확인한다.

### ④ Preservation Check — 회귀 방지
- ⓪에서 나열한 **변경 금지 영역**과 실제 코드 변경 범위를 교차 검증한다.
- 이번 변경으로 인해 **변경하지 않은 기존 동작**이 달라지는 경로가 있는지 추적한다.
- 달라지는 경로가 발견되면 코드를 수정하거나, 불가피하면 작업지시서에 부작용을 명시한다.

### ⑤ Faithfulness Check — 반영 확인
- ①~④에서 발견한 모든 결함이 최종 코드에 실제로 반영되었는지 확인한다.
- 누락된 수정이 있으면 코드를 갱신한다.

### ⑥ Step Sequence Check — 작업지시서 전체 정합성
- 전체 Step을 역순으로 훑어, 앞 Step의 시그니처·반환값·상태 변경이 뒤 Step에서 참조될 때 불일치가 없는지 확인한다.
- 새로 추가한 요소(함수·패널·변수)가 다른 Step에서 올바르게 연결·해제되는지 확인한다.

### 출력 규칙

작업지시서의 영향 범위 분석 섹션에 아래 항목을 필수 포함한다.

- ① Dry Run: [대표 입력] → [추적 요약 1~2줄] → [결과]
- ② 교훈 대조: [해당 L-xx → 반영 방법] 또는 [전수 대조 완료, 해당 없음]
- ③~⑥ 결함: [발견 시 내용 기술] 또는 ✅ 없음

이 항목이 포함된 작업지시서만 ✅ POLT로 간주한다.

### 반복 규칙
- 결함 발견 시 코드 수정 후 ①~⑤를 재수행한다. ⑥은 최종 1회만 수행.
- **최대 2회 반복**. 3회차에도 결함이 남으면 해당 Step에 `⚠️ POLT 미해결 — [결함 내용]`을 표기하여 사용자에게 보고한다.

## 8. 파일 업로드 검증

사용자가 파일을 업로드하면 첫 줄의 `PROJECT:` 주석으로 프로젝트를 확인한다.
주석이 없으면 파일 내 핵심 함수/변수로 판별한다.
현재 작업 프로젝트와 불일치하면 즉시 중단하고 사용자에게 확인한다.

## 9. 디버깅 프로토콜

### 1단계 — AI 자체 해결 (사용자 개입 0)
파일 구조에서 관련 파일 특정 → GitHub raw URL로 소스 직접 읽기 → 가설 수립 → 수정 코드 특정 → 작업지시서 출력.

### 2단계 — 1회 요청 (사용자 개입 1회)
조건: 런타임 상태를 알 수 없을 때. 콘솔 명령어를 한 번에 전부 제시. 예상 결과 포함.

### 3단계 — 브라우저 조작 (최후 수단)

### 요청 규칙
- 소스코드로 먼저 추론. 추론으로 특정되면 스크린샷/콘솔 없이 작업지시서 출력.
- 콘솔 요청 금지: 함수 존재 여부, CSS 선택자, HTML 구조, 전역 변수 선언, 코드 로직.
- 콘솔 필요: LocalStorage 실제 데이터, 런타임 변수 값, computed style, JS 에러, 네트워크 결과.

## 10. 소스 참조 공통 규칙

### 참조 우선순위
1순위 — 사용자 업로드 파일 (push 안 한 변경분 우선)
2순위 — GitHub raw URL 크롤링
3순위 — 사용자에게 업로드 요청 (1, 2 불가 시에만)

### 크롤링 규칙
- 소스 크롤링 전에 AGENTS.md의 "크롤링 제외" 목록을 확인한다. 제외 파일은 크롤링하지 않고 업로드를 요청한다.
- 크롤링 실패(잘림/빈 응답) 시: 사용자에게 업로드를 요청한다. 잘린 파일 기반으로 작업지시서를 작성하지 않는다.
- 크롤링 제외 목록의 관리는 수동 일괄 점검으로 수행한다. 크롤링 실패 시 자동 등록하지 않는다.
- 한 세션 최대 4파일

### 혼합 참조 주의
- 직전 작업지시서가 수정한 파일은 크롤링 대신 업로드 요청
- 파일이 잘렸으면 업로드 요청

### 컨텍스트 압축 발생 시
- 사용자에게 알림 + 재업로드 요청
- 기억 의존 금지, GitHub raw로 재확인

### Progressive Disclosure 상세

**대전제:** 사용자는 세션 시작 시 opus.md만 업로드한다. 나머지 모든 운영 문서는 GitHub raw URL로 크롤링 가능해야 한다.

## 13. 로컬 백업

앱 DB의 로컬 백업은 `C:\dev\backups\{프로젝트명}\`에 저장한다. Git 레포 밖이므로 `.gitignore` 불필요. 새 앱에 백업을 추가할 때 이 경로를 따른다.

## 14. 프로세스 안전 규칙 (Process Safety)

1. **프로세스 종료 명령 사용 금지**: `Stop-Process`, `taskkill`, `kill`, `pkill` 등 프로세스 종료 명령을 작업지시서 및 실행 스크립트에 포함하지 않는다.
2. **GUI 앱 실행/종료는 사용자 수동**: GUI 실행파일의 시작과 종료는 사용자가 직접 수행한다. 작업지시서에는 `사용자: [앱 이름]을 종료해주세요` 형태의 안내문만 포함한다.
3. **시스템 프로세스 이름 충돌 주의**: 빌드 출력물 이름이 시스템 프로세스명(explorer, svchost, csrss, dwm, lsass 등)과 동일한 경우, 프로세스 관련 명령을 절대 사용하지 않는다.
4. **빌드 전 앱 종료 필요 시**: `사용자: dist 폴더의 앱이 실행 중이면 종료해주세요`라고 안내하고, 사용자 확인 후 빌드를 진행한다.
