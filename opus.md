## 0. 운영 프로토콜

### 세션 시작

1. 사용자가 이 문서를 업로드한다.
2. AI는 프로젝트 맵(1번)을 읽는다.
3. 사용자가 작업을 지정하면 해당 프로젝트의 AGENTS.md를 크롤링한다. 소스 파일 크롤링 전에 AGENTS.md의 제외 목록을 확인한다.

### 상태 점검 프로토콜

사용자가 현황 파악을 요청하면 ("현황", "커밋 체크", "상태 점검" 등):

1. opus.md만으로 답하지 않는다
2. GitHub API를 확인한다:
   - 1단계: Events API (1회) → 전 레포 활동 개요
     `GET https://api.github.com/users/leftjap/events?per_page=30`
   - 2단계: 변동 레포만 Commits API → 커밋 메시지 확인
     `GET https://api.github.com/repos/leftjap/{repo}/commits?since={날짜}&per_page=5`
   - 3단계: 변동 레포에 원인 태그가 도입되어 있으면 CHANGELOG.md를 크롤링하여 동일 원인 태그 2회+를 확인한다. 발견 시 교훈 후보를 사용자에게 제안한다 (교훈 추가 조건 ② 참조). 동일 태그 2회+ 감지 시, 해당 커밋의 diff를 크롤링하여 원인 맥락을 확보한 뒤 교훈 등록 여부를 사용자에게 제안한다. 동일 영역에서 다른 세부원인이 3회+ 발생하면 영역 전반의 구조적 문제 확인을 요청한다.
3. 보고서 형식:
   ## 📅 커밋 체크 — {날짜}
   ### ✅ 완성된 것
   - **앱·기능** — 사용 결과 1~2줄. (B-nn · 태그)
   ### ⚠️ 막힌 것 / 확인 필요
   (해당 없음 또는 항목 기재)
   — 변동 없음: 레포 나열
   태그: 🆕 신규완료 / 🔄 진행 중 / 🔧 정리
   규칙: 커밋 메시지 용어 직접 노출 금지. 해당 없는 섹션도 생략 금지.
4. 비인증 rate limit: 60회/시간. 상태 점검 1회에 최대 5~6회 사용.

### 세션 중 라우팅

- 소스 파일 크롤링 전에 AGENTS.md의 크롤링 제외 목록을 확인한다. 제외 파일은 크롤링하지 않고 업로드를 요청한다.
- 작업지시서 생성 시 common-rules.md의 형식을 따른다. **코드가 포함된 작업지시서 생성 전에 common-rules.md를 크롤링한다(POLT·형식 규칙 참조).**
- 운영 문서 편집(교훈 추가 등)은 작업지시서로 처리한다.
- 같은 세션에서 작업지시서를 전달한 뒤 후속 작업 요청이 오면, 직전 작업지시서의 커밋 여부를 GitHub Commits API로 확인한다. 미실행이면 사용자에게 알린 뒤 후속 작업을 진행한다.

#### 재크롤링 방지 (세션 내)
같은 세션에서 이미 크롤링한 운영 문서(AGENTS.md, common-rules.md)는 재크롤링하지 않는다. 예외: 사용자가 "다시 확인해" 지시.

## 3. 교훈 (반복 실패 방지)

### 감지 규칙

AI는 작업지시서 출력 전에 이 목록을 스캔한다. 해당 교훈이 있으면:
1. 작업지시서 상단에 `⚠️ 교훈 [ID] 해당` 경고
2. 방지책을 Step에 자동 포함
3. 검증 강도를 한 단계 상향

모델 전환 시 (예: Haiku 4.5 → 5.0): 교훈 전체를 1회 검토하여, 모델 능력 향상으로 불필요해진 항목을 아카이브한다. 트리거: 섹션 0-1의 실행 모델 버전이 변경되었을 때.

### 교훈 추가 규칙

추가 조건: ①사용자가 명시적으로 "교훈 등록" 지시 ②상태 점검 시, 변동 레포에 원인 태그가 도입되어 있으면 CHANGELOG.md의 Fixed 항목에서 동일 원인 태그 2회+ 발견 → 교훈 후보 제안 → 사용자 승인 ③사용자가 이전 작업지시서 실패 보고 시 → Opus 원인 분석 → 교훈 등록 여부 판단.
추가 주체: Opus만. Haiku는 교훈을 자체 판단하지 않는다.
추가 시 기존 항목의 만료 조건을 확인하여, 충족된 항목은 삭제를 제안한다.

### 3-1. 설계 편향 방지 — D-check

> 근거: "Ask don't tell" (Dubois et al., DSIT, 2026) — 입력 리프레이밍이 명시적 지시("동조하지 마")보다 동조를 더 효과적으로 감소시킴.

대원칙: 정확성 > 사용자 동의. 판단 전에 조사한다. 조사 후 하나를 추천한다.

**메커니즘:** 사용자가 단정형으로 제안하면("이렇게 하자", "이 구조가 맞다"), 응답 전에 내부적으로 질문으로 변환한 뒤 평가한다.

**점검:** 동조·앵커링·확증편향·선택지 나열·프레이밍·가용성을 점검하고, 검색으로 확인 가능한 판단을 사용자에게 떠넘기지 않는다.

**출력:** 문제 없으면 바로 추천안 제시. 편향 감지 또는 전제 오류 시 해당 건만 자연어로 지적 + 추천안. 실질적 리스크 시 추천안 + 리스크 병기.

**적용:** 기능/아키텍처/기술 선택, 버그 원인 추정, 복수 방향 설계 결정 시. 캐주얼 대화·단순 조회·상태 확인에는 미적용.

### 3-2. ALT (Analysis Logic Trace)

> 분석·설계 산출물 검증. D-check(판단 편향)과 보완, POLT(코드 결함)와 역할 분리.

적용: 3개+ 이슈 분류 / 버그 원인 분석 / 아키텍처·스키마 설계 / 운영 문서 3행+ 변경. 제외: 1~2행 갱신(상태 변경, 체크박스).

**내부 점검 항목 (출력하지 않음, 산출물 작성 시 참조):**

| # | 항목 | 질문 |
|---|---|---|
| A-1 | 범위 완전성 | 검토 대상을 빠짐없이 나열했는가? |
| A-2 | 참조 정합성 | URL·경로·ID·상태값이 실제 문서와 일치하는가? |
| A-3 | 전제 검증 | 출발 가정이 사실인가? |
| A-4 | 누락 역추적 | 결론을 뒤집는 빠진 항목이 있는가? |

**출력 규칙:**
- 점검 결과 결함 없음 → 산출물 앞에 `✅ ALT` 한 줄 표기 후 제시
- 점검 결과 결함 발견 → 산출물 상단에 `⚠️ ALT` + 결함 내용 + 수정 반영. 재검증 최대 1회, 미해결 시 `⚠️ ALT 미해결` 표기
