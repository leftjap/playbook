# common-rules.md — 전 프로젝트 공통 작업 규칙

> 이 문서는 모든 프로젝트의 AGENTS.md에서 참조하는 공통 규칙이다.
> 프로젝트별 고유 규칙은 각 AGENTS.md에 있다.
> MUST/MUST NOT은 RFC 2119 의미로 사용한다.

---

## 1. 트랙 판단

### 트랙 A — 즉시 진행

조건 (모두 충족):
- 요청이 명확하다 (무엇을 어떻게 바꿀지 특정 가능)
- 해법이 하나뿐이다 (선택지/트레이드오프 없음)
- 영향 범위가 좁다 (1~2개 파일, 고위험 함수 미포함)

→ 영향 범위 분석 후 바로 작업지시서 출력.

### 트랙 B — 방향 확인 후 진행

조건 (하나라도 해당):
- 해법이 여러 개이고 선택이 필요하다
- 요청이 모호하거나 의도 확인이 필요하다
- 영향 범위가 넓다 (3개+ 파일, 고위험 함수 포함)
- 기존 동작이 바뀔 수 있다

→ 방향 확인서 출력 → 사용자 승인 → 작업지시서 출력. 승인 후 재확인하지 않는다.

### 판단 규칙
- 트랙 A면 `[트랙 A]`, 트랙 B면 `[트랙 B]` 표기
- 판단이 애매하면 트랙 B
- 사용자가 "바로 만들어" 등 즉시 진행을 명시하면 트랙 A로 전환

### 방향 확인서 형식
요청 이해 / 원인 분석(버그 시) / 해결 방향 / 영향 범위 / 관련 규칙 / 대안

## 2. 작업지시서 형식

⚠️ 모든 Step을 빠짐없이 순서대로 실행하세요. 특히 마지막 커밋 Step을 절대 생략하지 마세요.

### 필수 섹션
- 고정 문구: ⚡즉시 실행(질문 금지) / 프로젝트 경로(절대) / 기존 파일 전제(새 파일 시 제거)
- 작업 유형: 기능 추가 / 버그 수정 / 정리
- 백로그 컨텍스트 (백로그 연결 작업일 때 필수):
  - 대상: B-xx (프로젝트 · 작업명)
  - 이 작업지시서가 완료하는 항목: (backlog.md 완료 조건 중 어떤 체크박스를 충족하는지)
  - 완료 후 backlog.md 갱신: (해당 체크박스 [x] 처리 + 현재 상태 문구 업데이트)
- 영향 범위 분석: 수정 대상 파일, 수정 금지, 영향 받는 전역 변수/함수, 고위험 함수 수정 여부, POLT 검증 (① Dry Run, ② 교훈 대조, ③~⑤ 결함), 변경 후 확인할 기존 동작
  범위 외 대응: "📋 발견" 또는 "⛔ 블로킹" 형식으로 기록. 범위 외 파일 수정 금지
- Step별: 파일(절대 경로), 위치(함수명), 작업, 교체 코드(함수 전체), 영향 받는 함수, 완료 확인
- Step N-1: 운영 문서 갱신 — 대상: opus.md(해당 시), backlog.md(해당 시), CHANGELOG.md(필수). 파일: `C:\dev\opus\opus.md`, `C:\dev\opus\backlog.md`, 프로젝트별 `CHANGELOG.md`. 다른 레포에서 실행 중일 때 절대 경로 필수.
- Step N: 커밋 & 푸시
- ⛔종료(추가 작업·질문 금지)
- 최종 확인 (사용자 수동) — 핵심 1~2개 + 회귀 1개, 합계 3개 이하. 형식: `[ ] [동작] → [기대 결과] (핵심/회귀)`

## 3. 작업 유형별 규칙

**기능 추가**: 새 함수 전에 기존 함수 재사용 가능 여부 확인. 새 CSS 추가 시 같은 선택자 존재 여부 먼저 검색. !important 금지.
**버그 수정**: 수정 전에 원인을 먼저 설명. 새 규칙 덮어쓰기 금지. 잘못된 선언을 직접 수정.
**정리(리팩토링)**: 동작 변경 없음을 확인한다.

## 4. Claude Code 실행 규칙

- MUST: 한 Step에 한 파일, 한 가지 변경
- MUST: 함수 수정 시 함수 전체를 교체 코드로 제공. 부분 스니펫 비교 금지
- MUST: CSS 수정 시 선택자 블록 전체 제공
- MUST: 모든 파일 경로는 절대 경로
- MUST: 기존 함수 수정 시 함수명과 현재 동작을 Step에 명시
- MUST: 각 Step에 완료 확인 조건 포함
- MUST: 마지막 Step에 커밋 & 푸시
- MUST: 커밋 메시지 형식 — 백로그 연결 작업: `B-xx: [타입]: [요약]` / 백로그 무관 작업: `[타입]: [요약]` (feat/fix/chore/refactor)
- MUST: 영향 범위 분석에 "수정 대상 파일"과 "수정 금지" 명시
- MUST: "변경 후 확인할 기존 동작" 체크리스트 포함
- MUST: 백로그 연결 작업의 마지막 커밋 전에 backlog.md의 해당 체크박스를 [x]로 갱신하고, "현재" 상태 문구를 업데이트한다
- MUST: 백로그 항목의 모든 완료 조건이 충족되면, 해당 항목을 backlog-archive.md로 이동한다
- MUST: "최종 확인 (사용자 수동)" 섹션이 비어 있으면 안 된다. Opus가 작업지시서 생성 시 채운다
- MUST NOT: "적절히", "비슷하게" 등 모호한 표현
- MUST NOT: sed, tr 등 Unix 텍스트 처리 도구 (Windows 환경)

### 자동 테스트 규칙 (Regression Guard)

적용 조건: 프로젝트 루트에 `package.json`이 있고 `scripts.test`가 정의된 프로젝트.

#### 작업지시서 생성 시 (Opus)
- MUST: 커밋 Step 직전에 `npm test` 실행 Step을 포함한다. 이 Step의 완료 확인은 "전체 테스트 PASS"이다.
- MUST: 테스트 실행 Step에 "실패 시 커밋하지 않고 아래 형식으로 보고 후 중단"을 명시한다.

#### 테스트 실패 시 행동 (Haiku)
- MUST: 즉시 작업을 중단한다. 이후 Step을 실행하지 않는다.
- MUST: 아래 형식으로 보고한다:
  ```
  ❌ 테스트 실패 — 커밋 중단
  실패 테스트: [테스트명 1], [테스트명 2], ...
  실패 원인 추정: [1줄 요약]
  변경한 파일: [파일 목록]
  ```
- MUST NOT: 테스트 파일(__tests__/, *.test.*, *.spec.*)을 수정하여 테스트를 통과시키려 시도한다.
- MUST NOT: `--no-verify`, `-n` 플래그로 pre-commit hook을 우회한다.
- MUST NOT: 테스트를 비활성화(skip, disable, 삭제)한다.
- MUST NOT: 실패한 테스트의 기대값을 변경한다.
- MAY: 자신이 이번 작업에서 수정한 코드(테스트 파일 제외)에 명백한 오류가 있으면 1회 수정 후 `npm test`를 재실행할 수 있다. 재실행도 실패하면 보고 후 중단한다.

### 범위 통제 규칙 (Scope Control)

- MUST: 영향 범위 분석의 "수정 대상 파일" 목록에 없는 파일은 수정하지 않는다
- MUST: 작업 실행 중 범위 외 버그를 발견하면 수정하지 않는다. 해당 Step의 완료 확인 아래에 `📋 발견: [파일명] — [증상 1줄]`을 기록하고 현재 작업을 계속 진행한다
- MUST: 범위 외 버그가 현재 작업을 블로킹하는 경우, 수정을 시도하지 않고 `⛔ 블로킹: [파일명] — [증상] — 이 버그 해결 필요`를 기록한 뒤 작업을 중단하고 사용자에게 보고한다
- MUST: 하나의 작업지시서에 "기능 추가"와 "버그 수정"을 혼합하지 않는다. 작업 중 발견한 버그는 별도 작업지시서로 처리한다
- MUST: 기존 테스트 파일(__tests__/, *.test.*, *.spec.*)은 수정 금지. 새 테스트만 추가 가능 (테스트 인프라 도입 후 적용)

#### 변경 금지 속성
CSS 보호 속성(`B-57 PROTECT` 주석) 및 상태 변수: 각 프로젝트 AGENTS.md 보호 체크리스트 참조. 삭제·값 변경 시 iOS PWA 실기기 검증 필수.

### 반복 실패 방지
- 동일 증상 fix/style 커밋 2회 연속 실패 시 3회차 전 사용자에게 경고: "⚠️ 컨텍스트 오염 확률이 높습니다. 다른 세션에서 시도해 보세요."
- 3회 연속 실패 시 즉시 중단, 시도 내역·실패 원인·다음 방향을 정리하여 보고

## 5. 갱신 규칙

### AGENTS.md
코드 구조를 기재하지 않으므로 코드 변경에 따른 갱신은 불필요.
갱신 필요: 새 파일 추가 시 파일 구조에 추가, 고위험/중위험 함수 목록 변경 시, 운영 규칙 자체 변경 시.

### CLAUDE.md
AGENTS.md 고유 주의사항이 변경되면 해당 프로젝트 CLAUDE.md도 갱신한다. 작업지시서에 AGENTS.md 갱신 Step이 포함될 때 CLAUDE.md 갱신 필요 여부를 확인하고, 필요하면 Step을 추가한다.

### 운영 문서 갱신
모든 작업지시서의 마지막 커밋 Step 직전에 운영 문서 갱신 Step을 포함한다. 절대 생략하지 않는다.
갱신 대상: opus.md(교훈 추가 시), backlog.md(백로그 연결 작업 시), CHANGELOG.md(모든 작업지시서).
- opus.md 파일: C:\dev\opus\opus.md
- opus.md 크롤링: https://raw.githubusercontent.com/leftjap/opus/main/opus.md
- opus.md·backlog.md 갱신 불필요: 백로그 외 사소한 수정(CSS만 변경, 오타 수정 등). CHANGELOG는 이 경우에도 기록한다.

### CHANGELOG 갱신

모든 작업지시서의 마지막 커밋 Step 직전에 해당 프로젝트의 CHANGELOG.md를 갱신한다.

- 형식: [Keep a Changelog](https://keepachangelog.com/en/1.1.0/) — Added / Changed / Fixed / Removed
- 날짜 헤더: `## YYYY-MM-DD`. 같은 날짜 헤더가 이미 있으면 해당 섹션에 추가. 없으면 파일 최상단(기존 날짜 위)에 새 헤더 생성
- 항목 형식: `- 한 줄 요약 — 부연 설명. (변경 파일)` 또는 `- 한 줄 요약 (B-xx)`
- 큰 결정(아키텍처·스키마·규칙 체계 변경)은 Changed 항목에 이유 1문장을 덧붙인다
- CHANGELOG.md가 크롤링 한계(10,000 토큰)에 근접하면 현재 연도분만 유지하고 이전 연도를 `CHANGELOG-{연도}.md`로 분리한다

### GAS 배포
Code.js 수정 시 반드시 clasp push Step을 포함한다.
- MUST: 기존 배포를 업데이트한다 (@HEAD 또는 기존 배포 ID에 새 버전 지정). URL이 유지된다.
- MUST NOT: 새 배포(Deploy → New deployment)를 생성한다 — URL이 변경되어 모든 클라이언트의 GAS_URL 수정이 필요해진다.
- MUST: GAS 배포 성공 후 Git 커밋. GAS 실패 시 클라이언트 코드도 푸시하지 않는다.
- MUST NOT: 작업지시서에 "Google Apps Script 에디터에서 수동 재배포" 안내를 포함한다. GitHub Actions auto-deploy(B-63)가 처리한다.

## 6. AI 응답 규칙

- 확신 수준: "확실합니다" / "높은 확률이지만 검증 필요" / "추측입니다".
- 검증 강도 "강화" 이상에서는 코드에 문제가 있다고 가정하고 찾는다.
- 사용자에게 콘솔 명령 실행을 요청하기 전에 코드 정적 분석으로 원인을 먼저 추정한다.

## 6-1. 코드 품질 사전 검증 — POLT (Pre-Output Logic Trace)

> 작업지시서에 코드 블록을 포함하기 전에 수행하는 검증. 결과를 영향 범위 분석에 필수 출력.

### 적용 조건
작업지시서 Step이 코드를 생성·교체·수정할 때.

### 절차

| # | 단계 | 핵심 행동 |
|---|---|---|
| ⓪ | Scope Lock | 변경 파일·함수·변수·CSS 나열. 호출자 역추적(시그니처·반환값·공유 상태 변경 시). 변경 금지 영역 나열(AGENTS.md ⛔ 자동 포함). 이후 ①~⑤는 변경 대상만을 범위 |
| ① | Dry Run | 대표 입력 1~2개, 라인별 변수 추적, 모든 분기 1회+ 통과, 루프 내 I/O 성능 표시 |
| ② | Self-Critique | 교훈 L-xx 전수 대조(의무) → 엣지케이스(null, undefined, 빈 배열, 빈 문자열, 0, 음수, 경계값) → 환각 객체 → 보안 → 중복 → 과잉 방어 → 유지보수성 |
| ③ | Wiring Check | 새 요소의 호출·참조 경로 확인. 기존 caller 동작 보존 확인. 공유 상태(전역, DOM, CSS, LS, GAS) 영향 확인. 크롤링 제외 caller는 ⚠️ 표기 |
| ④ | Preservation | 변경 금지 영역(⓪)과 실제 변경 교차 검증. 기존 동작이 달라지는 경로 추적 |
| ⑤ | Faithfulness | ①~④ 결함이 최종 코드에 반영되었는지 확인 |
| ⑥ | Step Sequence | 전체 Step 역순 정합성. 새 요소 연결·해제 확인. 최종 1회만 |

### 출력 규칙
영향 범위 분석에 필수 포함:
- ① Dry Run: [입력] → [추적 요약] → [결과]
- ② 교훈 대조: [해당 L-xx → 반영] 또는 [전수 대조 완료, 해당 없음]
- ③~⑥ 결함: [내용] 또는 ✅ 없음

### 반복 규칙
결함 발견 시 코드 수정 후 ①~⑤ 재수행. ⑥은 최종 1회. **최대 2회 반복.** 3회차에도 잔존 시 `⚠️ POLT 미해결 — [내용]` 표기.

## 8. 파일 업로드 검증

사용자가 파일을 업로드하면 첫 줄의 `PROJECT:` 주석으로 프로젝트를 확인한다.
주석이 없으면 파일 내 핵심 함수/변수로 판별한다.
현재 작업 프로젝트와 불일치하면 즉시 중단하고 사용자에게 확인한다.

## 9. 디버깅 프로토콜

| 단계 | 조건 | 행동 |
|---|---|---|
| 1 | 소스로 특정 가능 | 크롤링→가설→작업지시서. 사용자 개입 0 |
| 2 | 런타임 상태 필요 | 콘솔 명령 일괄 제시 + 예상 결과. 사용자 개입 1회 |
| 3 | 최후 수단 | 브라우저 조작 |

소스로 먼저 추론. 특정되면 스크린샷/콘솔 없이 작업지시서 출력.
콘솔 요청 금지: 함수 존재, CSS 선택자, HTML 구조, 전역 변수, 코드 로직.
콘솔 필요: LS 데이터, 런타임 값, computed style, JS 에러, 네트워크.

## 10. 소스 참조 공통 규칙

### 참조 우선순위
1순위 — 사용자 업로드 파일 (push 안 한 변경분 우선)
2순위 — GitHub raw URL 크롤링
3순위 — 사용자에게 업로드 요청 (1, 2 불가 시에만)

### 크롤링 규칙
- 소스 크롤링 전에 AGENTS.md의 "크롤링 제외" 목록을 확인한다. 제외 파일은 크롤링하지 않고 업로드를 요청한다.
- 크롤링 실패(잘림/빈 응답) 시: 사용자에게 업로드를 요청한다. 잘린 파일 기반으로 작업지시서를 작성하지 않는다.
- 크롤링 제외 목록의 관리는 수동 일괄 점검으로 수행한다. 크롤링 실패 시 자동 등록하지 않는다.
- 한 세션 최대 4파일

### 혼합 참조 주의
- 직전 작업지시서가 수정한 파일은 크롤링 대신 업로드 요청
- 파일이 잘렸으면 업로드 요청

### 컨텍스트 압축 발생 시
- 사용자에게 알림 + 재업로드 요청
- 기억 의존 금지, GitHub raw로 재확인

## 13. 로컬 백업

앱 DB의 로컬 백업은 `C:\dev\backups\{프로젝트명}\`에 저장한다. Git 레포 밖이므로 `.gitignore` 불필요. 새 앱에 백업을 추가할 때 이 경로를 따른다.

## 14. 프로세스 안전 규칙 (Process Safety)

1. **프로세스 종료 명령 사용 금지**: `Stop-Process`, `taskkill`, `kill`, `pkill` 등 프로세스 종료 명령을 작업지시서 및 실행 스크립트에 포함하지 않는다.
2. **GUI 앱 실행/종료는 사용자 수동**: GUI 실행파일의 시작과 종료는 사용자가 직접 수행한다. 작업지시서에는 `사용자: [앱 이름]을 종료해주세요` 형태의 안내문만 포함한다.
3. **시스템 프로세스 이름 충돌 주의**: 빌드 출력물 이름이 시스템 프로세스명(explorer, svchost, csrss, dwm, lsass 등)과 동일한 경우, 프로세스 관련 명령을 절대 사용하지 않는다.
4. **빌드 전 앱 종료 필요 시**: `사용자: dist 폴더의 앱이 실행 중이면 종료해주세요`라고 안내하고, 사용자 확인 후 빌드를 진행한다.
