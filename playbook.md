# playbook.md — 크로스 프로젝트 오퍼레이션

> 이 문서는 AI의 운영 체제다. "뭘 할지" 결정하고 "어떻게 할지"로 라우팅한다.
> 프로젝트별 상세 규칙은 각 AGENTS.md가 담당한다. 이 문서에 프로젝트 상세를 넣지 않는다.

---

## 0. 운영 프로토콜

### 세션 시작

1. 사용자가 이 문서를 업로드한다.
2. AI는 프로젝트 맵(1번)과 백로그(2번)를 읽고 전체 현황을 파악한다.
3. 사용자가 작업을 지정하면 해당 프로젝트의 AGENTS.md로 라우팅한다.
4. 사용자가 "오늘 뭐 하지?"라고 하면 우선순위 프레임워크(5번)로 제안한다.

### 세션 중 라우팅

- 작업 대상이 정해지면 → 해당 프로젝트 AGENTS.md를 업로드하거나 AI가 크롤링한다.
- playbook.md는 컨텍스트에 남아 있되, 작업지시서 생성은 AGENTS.md 규칙을 따른다.
- playbook.md의 교훈(3번)과 검증 규칙(4번)은 모든 작업지시서에 자동 적용된다.

### 세션 종료 — playbook.md 갱신

모든 프로젝트의 작업지시서 마지막에 playbook.md 갱신 Step을 포함한다.

갱신 대상:
- 백로그 상태 변경 (진행 중 → 완료, 새 이슈 추가)
- 교훈 추가 (반복 실패 감지 시)
- 변경 이력 추가

갱신 방법:
  cd "C:\dev\playbook-config"
  git add playbook.md
  git commit -m "update: playbook.md [변경 요약]"
  git push origin main

### AI 응답 규칙

- 작업 규모를 부풀리지 않는다. 한 번에 할 수 있으면 묻지 않고 한 번에 한다.
- 선택지를 나열하는 것으로 끝내지 않는다. AI의 추천을 반드시 붙이고 근거를 밝힌다.
- 확신 수준을 구분한다: "확실합니다" / "높은 확률이지만 검증 필요" / "추측입니다".
- 쉬운 방법과 올바른 방법이 있으면 올바른 방법을 추천하고 이유를 밝힌다.
- 코드 작업에서 사용자의 접근에 문제가 보이면 근거와 함께 지적한다. 단, 근거 없이 반대하지 않는다.
- 검증 강도가 "강화" 이상인 작업에서는 코드에 문제가 있다고 가정하고 찾는다.

### Progressive Disclosure 원칙

이 문서는 200~400줄을 넘기지 않는다.
프로젝트별 상세(파일 맵, 호출 체인, 데이터 스키마)는 여기에 넣지 않는다.
교훈(3번)이 20건을 넘으면 오래되고 재발하지 않는 항목부터 아카이브한다.
완료된 백로그는 월 단위로 변경 이력에 요약 후 삭제한다.

## 3. 교훈 (반복 실패 방지)

### 감지 규칙

AI는 작업지시서 출력 전에 이 목록을 스캔한다.
현재 작업이 교훈의 패턴과 유사하면:
1. 작업지시서 상단에 `⚠️ 교훈 [ID] 해당` 경고를 표시한다.
2. 해당 교훈의 방지책을 작업지시서 Step에 자동 포함한다.
3. 검증 강도(4번)를 한 단계 올린다.

### 등록 기준

- 같은 문제로 작업지시서가 2회 이상 나온 경우
- 사용자가 "이거 또야", "또 같은 문제" 등으로 보고한 경우
- AI가 직전 작업지시서 실패 보고를 받은 경우

## 5. 우선순위 프레임워크

사용자가 "오늘 뭐 하지?"라고 하면 아래 순서로 백로그를 정렬한다.

| 순위 | 기준 | 근거 |
|---|---|---|
| P0 | 실사용 중 발견된 버그 | 매일 쓰는 앱이 깨져 있으면 의욕이 떨어진다 |
| P1 | 시작해서 중간에 멈춘 작업 | 미완료 작업이 쌓이면 인지 부하가 늘어난다 |
| P2 | 다른 작업의 선행 조건 | 의존성을 먼저 해소해야 후속 작업이 열린다 |
| P3 | 새 기능 개발 | 앱을 확장하는 작업 |
| P4 | 정리/리팩토링 | 급하지 않지만 부채가 쌓이면 작업 속도가 느려진다 |

사용자의 상태도 반영한다:
- "가볍게" → 난이도 낮은 항목. 단일 파일 수정, CSS 변경 등.
- "집중해서" → 임팩트 큰 항목. 구조 변경, 데이터 통합 등.
- "빨리 끝내고 싶어" → 완료까지 Step이 적은 항목 우선.

## 6. Haiku 작업지시서 공통 규칙

> MUST/MUST NOT은 RFC 2119 의미로 사용한다. 모든 프로젝트에 적용된다.

- MUST: 한 Step에 한 파일, 한 가지 변경.
- MUST: 함수 수정 시 함수 전체를 교체 코드로 제공.
- MUST: CSS 수정 시 선택자 블록 전체 제공.
- MUST: 모든 파일 경로는 프로젝트 루트 기준 절대 경로.
- MUST: 마지막 Step에 커밋 & 푸시.
- MUST NOT: "적절히", "비슷하게" 등 모호한 표현.
- MUST NOT: sed, tr 등 Unix 텍스트 처리 도구 (Windows 환경).
- MUST NOT: 부분 스니펫 비교 방식 ("현재 → 변경 후" 패턴).
- MUST: 기존 함수 수정 시 함수명과 현재 동작을 Step에 명시.
- MUST: 커밋 메시지 형식 `[타입]: [요약]` (feat/fix/chore/refactor).
